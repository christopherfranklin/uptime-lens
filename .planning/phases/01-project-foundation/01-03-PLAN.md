<task type="auto">
  <name>Task 1: Create Railway worker scaffold and deploy full stack</name>
  <files>worker/package.json, worker/tsconfig.json, worker/src/index.ts, worker/Dockerfile, vercel.json</files>
  <action>
1. Create `worker/` directory with a standalone Node.js project:

   `worker/package.json`:
   - name: "uptime-lens-worker"
   - scripts: "start": "node dist/index.js", "build": "tsc"
   - dependencies: @neondatabase/serverless, dotenv
   - devDependencies: typescript, @types/node

   `worker/tsconfig.json`:
   - target: ES2022, module: Node16, moduleResolution: Node16
   - outDir: "./dist", rootDir: "./src"
   - strict: true

   `worker/src/index.ts`:
   - A minimal long-running Node.js process
   - On startup: log "Uptime Lens Worker starting..."
   - Verify database connectivity: connect to Neon using DATABASE_URL, run a simple SELECT 1 query, log success or failure
   - Start a basic HTTP health check server on PORT (from env, default 3001): GET / returns { status: "ok", timestamp: ISO string, version: "0.1.0" }
   - Log "Worker ready" after health check server starts
   - Handle SIGTERM/SIGINT gracefully (log shutdown, close server, exit)
   - This is a PLACEHOLDER -- Phase 4 will add BullMQ and the actual check engine

   `worker/Dockerfile`:
   ```dockerfile
   FROM node:22-slim
   WORKDIR /app
   COPY package*.json ./
   RUN npm ci --production
   COPY dist/ ./dist/
   EXPOSE 3001
   CMD ["node", "dist/index.js"]
   ```

2. Run `cd worker && npm install && npm run build` to verify the worker compiles.

3. Create `vercel.json` in project root if needed for any custom configuration. For a standard Next.js 16 app, Vercel auto-detects the framework -- so this file may be minimal or unnecessary. Only create it if there is a specific need (e.g., rewrites, headers). If not needed, skip this file.

4. Deploy to Vercel:
   - Run `npx vercel --yes` to link the project and deploy (if Vercel CLI is available)
   - If Vercel CLI is not installed: `npm install -g vercel` then `npx vercel --yes`
   - Set environment variables in Vercel dashboard or via CLI:
     `vercel env add DATABASE_URL production` (user will need to provide the Neon production connection string)
     `vercel env add RESEND_API_KEY production`
     `vercel env add NEXT_PUBLIC_APP_URL production` (set to the Vercel deployment URL)
   - If the user has not yet set up Neon/Resend accounts, document what they need to do and pause for their input

5. Push schema to production Neon database:
   - Once DATABASE_URL points to production Neon: `DATABASE_URL=<prod_url> npx drizzle-kit push`
   - This creates all 7 tables in the production database

6. Deploy worker to Railway:
   - If Railway CLI is available: `railway init` in the worker directory, then `railway up`
   - If not: guide user to connect the GitHub repo to Railway and set the worker directory as the root
   - Set DATABASE_URL environment variable in Railway dashboard
   - Verify health check endpoint is accessible

Note: Deployment steps require the user to have accounts and credentials ready. If any service is not set up, create a checkpoint-style note explaining exactly what the user needs to do. Do not block on missing credentials -- set up everything possible and clearly document remaining manual steps.
