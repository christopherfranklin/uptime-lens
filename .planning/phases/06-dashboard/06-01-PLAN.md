<task type="auto">
  <name>Task 1: Install Recharts, create query modules and utilities</name>
  <files>package.json, lib/queries/monitors.ts, lib/queries/uptime.ts, lib/queries/charts.ts, lib/queries/incidents.ts, lib/utils.ts</files>
  <action>
1. Install Recharts: `npm install recharts@3.8.0`

2. Create `lib/utils.ts` with shared helper functions:
   - `formatDuration(ms: number): string` -- converts milliseconds to human-readable string (e.g. "2h 15m", "45s", "3d 2h"). Duplicate the ~10-line function from worker/src/emails/templates.ts rather than importing across package boundaries.
   - `formatResponseTime(ms: number | null): string` -- returns formatted response time (e.g. "142ms") or "--" when null.
   - `getRangeSince(range: string, now?: Date): Date` -- converts range string ("24h", "7d", "30d", "90d") to a Date object representing the start of that range. Default `now` to `new Date()`.

3. Create `lib/queries/monitors.ts` with two exports:
   - `getMonitorsWithStats(userId: string)` -- fetches all monitors for a user, then for each monitor fetches the latest heartbeat responseTimeMs and 24h uptime percentage (from heartbeatsHourly). Returns array of monitor objects enriched with `latestResponseTimeMs: number | null` and `uptime24h: number`. Use the N+1 approach (simple, fine for 5-20 monitors). Default uptime to 100 when no data (project convention). Use `COALESCE(SUM(...), 0)` for safety.
   - `getMonitorById(monitorId: number, userId: string)` -- fetches a single monitor by ID with ownership check (WHERE id = monitorId AND userId = userId). Returns the monitor or null. This will be consumed by Plan 02's detail page.

4. Create `lib/queries/charts.ts` with one export:
   - `getChartData(monitorId: number, range: string)` -- switches data source based on range: "24h" queries `heartbeats` (raw), "7d" queries `heartbeatsHourly`, "30d"/"90d" queries `heartbeatsDaily`. Returns array of `{ time: number; responseTime: number | null }` where `time` is a Unix timestamp (number, not Date -- avoids serialization issues between server and client components per Pitfall #8). Use `getRangeSince` for the time filter. Order by time ASC.

5. Create `lib/queries/uptime.ts` with one export:
   - `getUptimeForAllRanges(monitorId: number)` -- fetches uptime percentages for all 4 time ranges in parallel using Promise.all. For 24h/7d, query `heartbeatsHourly`. For 30d/90d, query `heartbeatsDaily`. Calculate as `SUM(successfulChecks) / SUM(totalChecks) * 100`, rounded to 2 decimal places. Default to 100% when totalChecks is 0. Returns `{ range: string; percentage: number }[]`.

6. Create `lib/queries/incidents.ts` with one export:
   - `getIncidentsForMonitor(monitorId: number, range: string, limit: number, offset: number)` -- queries incidents table filtered by monitorId and time range. Ongoing incidents (status = "ongoing") are always included regardless of start time. Orders ongoing first, then by startedAt DESC. Fetches `limit + 1` rows to determine `hasMore`. Returns `{ items: Incident[]; hasMore: boolean }`.

All query functions use `import { db } from "@/lib/db"` and the appropriate schema tables/enums. All use drizzle-orm query builders with `eq`, `and`, `gte`, `desc`, `asc`, `sql`, `or` as needed.
  </action>
